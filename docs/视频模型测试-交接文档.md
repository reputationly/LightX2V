**通用流程**:
```bash
scp scripts/smoke/test_model.sh edt-vpn:/data/         # + 需要的其他脚本
ssh edt-vpn
docker rm -f $(docker ps -aq) 2>/dev/null              # 清干净再跑
NAME=xx MODEL_CLS=.. TASK=.. MODEL_PATH=.. CFG=.. ... bash /data/test_model.sh
```

`test_model.sh` 已支持:单卡(`NP=1`→python)/多卡(`NP=4`→torchrun ulysses)、图像任务自动走 image 端点、首尾帧(`LAST_FRAME`)、音频(`AUDIO`)、`RESIZE_MODE=null`(720p)。**大多数新模型直接用它,只需配好 config + 喂对 env。**

## 5. 给新模型搭测试的标准套路

1. **找官方脚本** `scripts/<model>/run_*.sh` → 抄出 `model_cls` / `task` / `config_json` / 输入字段(image/audio/last_frame)
2. **拿官方配置** `configs/<model>/*.json` → 复制一份,把 DiT 指到 **int8 目录**(`dit_quantized:true` + `dit_quant_scheme:int8-torchao` + `*_quantized_ckpt`),需要多卡就加 `parallel:{seq_p_size:4,seq_p_attn_type:"ulysses"}`
3. **先 int8 单卡跑通**(`NP=1`)→ 再 4卡(`NP=4`)→ 再 bf16 对比画质
4. 用 `test_model.sh` 跑;图像任务它自动走 image 端点
5. **对每个模型记录**:加载/生成耗时、GPU峰值、(多卡)线性度、int8 vs bf16 画质,写进各自报告 md

## 6. 可复用的关键经验(Wan I2V 验出来的,大概率通用)

- **int8 的价值是显存,不是速度**:int8-torchao 在 A100/ARM **不走 INT8 tensor core**,单卡 int8 可能比 bf16 还慢;**提速靠多卡 ulysses**(切序列,线性加速 + 激活÷卡数)。
- **bf16 多卡在 256G 机器不可行**:ulysses 每 rank 复制整份 bf16 到 CPU,4×易超内存 CPU OOM。要多卡 → 先量化 int8。
- **MoE 模型 bf16 单卡必须 `cpu_offload:true` + `offload_granularity:"model"`**(一次换一个专家);用 `"block"` 会黑屏。
- **显存 = 权重 + 激活**:激活随分辨率/帧数涨且 **offload 搬不走**(前向必须在 GPU);加分辨率/时长 → 多卡(切激活),不是 offload。
- **Wan i2v 分辨率**:请求体传 `resize_mode:null` 才走 `max_area=target_h×w` 路径;`target_shape` 字段无效。"720p"=面积预算 ~92万像素,宽高跟图比例。
- **int8 量化敏感层**:converter 用原生 model_type 自动护住 embedder/norm;块级敏感(整块不量化)只有 LTX 需要,其他模型先全量化、崩了再 `--ignore-quant-keys` 跳块重转。
- **harness 判定**:`test_model.sh` 已改 status-first(只认 `completed`),产物 <50KB 判失败(防黑屏假绿)。
- **测对比要同图同分辨率**(否则像 480p flf2v 比 i2v"快一倍"是分辨率假象)。
