## 2.5 进行中 ⭐ SeedVR2 视频超分(当前任务,2026-06-28)

> 目标:AIGC 视频超分(720p→1080p / 480p→1080p)。python 详细跑法见 `docs/SeedVR2-超分-运行手册.md`;

### 权重(已下到 NFS ✅)
- 下载脚本已加标签:`MODELS="seedvr_3b seedvr_7b" bash /data/download_models.sh`
  —— 走**魔搭社区镜像** `bytedance-community/SeedVR2-3B|7B`(~50MB/s,快;无官方 ModelScope,HF 官方 `ByteDance-Seed/*` 作回退)
- `/nfs-data/models/ByteDance-Seed/SeedVR2-3B`(**16G**)= 全:`seedvr2_ema_3b.pth`(13.6G DiT)+ `ema_vae.pth` + `pos_emb.pt` + `neg_emb.pt` → **可直接跑**
- `/nfs-data/models/ByteDance-Seed/SeedVR2-7B`(**62G**)= **只有两个 DiT**:`seedvr2_ema_7b.pth`(普通)+ `seedvr2_ema_7b_sharp.pth`(锐化版),**无 vae/emb**(7B 仓库本就缺 emb)
- ⚠️ 跑 7B:`model_path` 指 **3B 目录**(借 vae+emb),config `dit_original_ckpt` 指 7B 某个 `.pth`;**切普通/锐化版只改这一行**,可同图对比。

### Docker/server 路线 = 可行(源码级已确认)
- runner `seedvr2` 已注册(`pipeline.py`);请求体 `VideoTaskRequest.video_path` 支持 SR;官方 `scripts/server/post_sr.py` 就是 server 走法(`POST /v1/tasks/video/` + `video_path`)。
- 起 server:`--model_cls seedvr2` + config `task=sr`,请求体传 `video_path`(**容器内本地路径**)。
- ⚠️⚠️ **sr_ratio 在 server 模式锁死 2.0**:请求体 schema 无 `sr_ratio` 字段、worker 不注入、config 也喂不进(runner 读 `input_info.sr_ratio`,默认 2.0)。
  - **720p→1080p**:2.0 够(被 target 1080p cap)→ **无需改代码**,直接跑。
  - **480p→1080p**:需 sr_ratio ≥ ~2.25,2.0 不够 → **必须给 `lightx2v/server/schema.py` 的 `VideoTaskRequest` 加 `sr_ratio` 字段才解锁**(1 行;链路 请求→`model_fields_set`→task_data→`update_input_info_from_dict`,`SRInputInfo` 本就有该字段)。
- 分辨率公式(`seedvr_runner.py:105`):`输出边长 = min(√(源h·w)·sr_ratio, √(target_h·target_w))`,再 `DivisibleCrop(16)`。480p(边长~640)×2.0=1280<1440(1080p),所以 480p 一步到 1080p 必须抬 sr_ratio。
- **harness 缺口**:`test_model.sh` 现在不发 `video_path`(只 image/last_frame/audio)→ 要加 `VIDEO_PATH` 透传。server 也不下载 video(无 `_process_video_path`)→ 视频必须挂进容器、用容器内路径。
- ✅ **不需要 decord**(runner 用 torchvision/PyAV,镜像已装),不用重建镜像。

### 关键参数 / 坑
- `infer_steps:1`(单步扩散,快);40G 必开 `cpu_offload:true`+`offload_granularity:model`+`use_tiling_vae:true`;`color_fix:gpu`;`sr_segment_length:81`/`sr_overlap:1`(长视频分段防 OOM)。
- ⚠️ 官方警告:对 **720p AIGC 这类轻退化输入易过度补细节 / 过锐化**。→ **锐化版 7B 风险更大,3B 可能更稳;本场景 7B 不一定更好**。先 3B 跑通看效果,再做 3B / 7B普通 / 7B锐化 三版对比。

1. **找官方脚本** `scripts/<model>/run_*.sh` → 抄出 `model_cls` / `task` / `config_json` / 输入字段(image/audio/last_frame)

## 6. 可复用的关键经验(Wan I2V 验出来的,大概率通用)

- **int8 的价值是显存,不是速度**:int8-torchao 在 A100/ARM **不走 INT8 tensor core**,单卡 int8 可能比 bf16 还慢;**提速靠多卡 ulysses**(切序列,线性加速 + 激活÷卡数)。
- **bf16 多卡在 256G 机器不可行**:ulysses 每 rank 复制整份 bf16 到 CPU,4×易超内存 CPU OOM。要多卡 → 先量化 int8。
- **MoE 模型 bf16 单卡必须 `cpu_offload:true` + `offload_granularity:"model"`**(一次换一个专家);用 `"block"` 会黑屏。
- **显存 = 权重 + 激活**:激活随分辨率/帧数涨且 **offload 搬不走**(前向必须在 GPU);加分辨率/时长 → 多卡(切激活),不是 offload。
- **Wan i2v 分辨率**:请求体传 `resize_mode:null` 才走 `max_area=target_h×w` 路径;`target_shape` 字段无效。"720p"=面积预算 ~92万像素,宽高跟图比例。
- **int8 量化敏感层**:converter 用原生 model_type 自动护住 embedder/norm;块级敏感(整块不量化)只有 LTX 需要,其他模型先全量化、崩了再 `--ignore-quant-keys` 跳块重转。
- **harness 判定**:`test_model.sh` 已改 status-first(只认 `completed`),产物 <50KB 判失败(防黑屏假绿)。
- **测对比要同图同分辨率**(否则像 480p flf2v 比 i2v"快一倍"是分辨率假象)。

**Z-Image 补充的通用经验(图像模型/新模型大概率通用):**
- **冷态 vs 热态差近一倍,必须测热态**:单发首张带 CUDA context + cuDNN/triton 首次 autotune,虚高(z_image 15s vs 热态 7.6s)。正确测法:**单容器连发 N 张、丢首张、取稳态均值**(见 `test_z_image_stress.sh`)。
- **autotune 是 per-shape(按分辨率),但缓存跨切换保留**:每个 H×W 首次"冷"(autotune),之后**永久缓存、即使穿插别的分辨率也不丢**(实测穿插重访,第二次冷掉到热)。所以扫分辨率耗时是冷态不可与热态混比;且**一个通用实例跑过所有分辨率各一次后全热** → 多实例生产**用通用池 + 启动预热全分辨率,别按分辨率绑实例**(后者招负载不均)。autotune 缓存在主机内存但只 KB 级;单卡实例主机内存仅 ~2GB(权重在显存),高并发只排队不增内存。
- **A100(Ampere)别用 Hopper kernel**:`attn_type=flash_attn3`、`fp8` 在 A100 跑不了/不加速 → 用 `sage_attn2`;`flash_attn3` 改 `sage_attn2`。
- **flashinfer 缺失坑**:`rope_type` 默认 `flashinfer`,本镜像没装 → 推理 `'NoneType' object is not callable`。改 `rope_type=torch`(纯 torch,零依赖)。**flux2 等用 flashinfer rope 的模型同样会踩。**
- **ulysses 卡数限制**:`seq_p_size` 必须**整除 attn head 数**(z_image 30 head → 只能 2/3/5/6,**4 卡报错**)。配多卡先查 head 数。
- **int8-torchao 是 weight-only**:不吃 INT8 算力,热态可能比 bf16 慢很多(z_image 慢 2.9×)。**显存够就用 bf16**;int8 只在显存吃紧(高分辨率/小卡)才值。判断标准:bf16 峰值离卡上限多远。
- **吞吐方案**:单图模型,多卡 ulysses 提速有限(z_image 仅 1.2×),**N×单卡实例 + 负载均衡**吞吐远高于少数多卡实例(z_image 4 单卡 0.530 img/s)。脚本 `test_z_image_4cards.sh`。
- **基准脚本要在"无有效结果"时非零退出**(防自动化把坏运行当成功)——`stress`/`sweep`/`4cards` 已加。
