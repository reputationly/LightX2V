# 视频生成平台 — 轻量自建方案设计(B 方向)

> 状态:设计稿(仅方案,未写代码)。最后更新:2026-06-25。
> 基于 LightX2V 自定义视频运行时,**不引入 GPUStack**;多节点/队列/网关全部轻量自建,最大化复用已验证的 LightX2V server + 容器。

---

## 1. 目标与需求

把现有"脚本出片"升级为一套**多节点视频生成平台**,要求:

1. **网页端**:页面配置(模型/拓扑/参数),配置完全来自网页,**不读本地 config 文件**(由网页生成临时 config)。
2. **模型覆盖**:LTX2.3(单卡 bf16)+ Wan2.2(int8,单卡/4卡 ulysses 720p)。
3. **多计算节点** + **任务队列** + **灵活动态加节点**。
4. **加节点挂 NFS**:NFS 内存放模型权重 + docker 镜像,新节点挂载即可取。
5. **产物入华为云 OBS**,用 OBS 生命周期规则做留存治理。
6. **对外 OpenAI 标准接口**(视频 + 图像),与网页端并存。

---

## 2. 总体架构

```
                         ┌──────────────────────────────────────────┐
  浏览器(运营/用户) ───▶│  Web 前端(配置页 + 生成页 + 产物预览)      │
  OpenAI 客户端 ────────▶│  Gateway / API 服务(FastAPI)              │
                         │   · OpenAI 兼容端点(图像/视频)            │
                         │   · 网页 API · 鉴权/Key · 入队             │
                         └───────────────┬──────────────────────────┘
                                         │ enqueue
                              ┌──────────▼──────────┐     ┌───────────────┐
                              │  Redis 队列 + 节点状态 │◀───▶│ Postgres       │
                              └──────────┬──────────┘     │ 节点/任务/Key/  │
                                         │ assign         │ 产物元数据      │
                              ┌──────────▼──────────┐     └───────────────┘
                              │  Dispatcher(调度器)  │
                              │  按模型/拓扑/空闲匹配 │
                              └───┬───────────┬──────┘
                          node1  │     node2 │   ...  nodeN     ← 每节点挂 NFS
                      ┌──────────▼──┐ ┌───────▼─────┐
                      │ Node Agent  │ │ Node Agent  │
                      │  ├ 注册节点  │ │             │
                      │  ├ 领取任务  │ │             │
                      │  ├ 调本地 API│ │             │
                      │  └ 传 OBS    │ │             │
                      │ LightX2V 容器│ │ LightX2V 容器│
                      │ (单卡/4卡)   │ │             │
                      └──────┬───────┘ └─────────────┘
                             │ 产物
                      ┌──────▼───────────────────────────────────┐
                      │  华为云 OBS(产物 + 生命周期规则)          │
                      └──────────────────────────────────────────┘
       NFS(共享):模型权重 + docker 镜像  ── 挂载到所有计算节点
```

**分层职责**

| 层 | 组件 | 职责 |
|----|------|------|
| 接入层 | Gateway/API(FastAPI) | OpenAI 兼容端点、网页 API、鉴权、入队;**唯一对外入口** |
| 调度层 | Redis(队列+节点状态) + Dispatcher | 全局队列、按"模型/拓扑/空闲"把任务派到节点 |
| 状态层 | Postgres | 节点注册表、任务记录、API Key、产物→OBS 映射 |
| 计算层 | Node Agent + LightX2V 容器 | 节点上跑推理(复用已验证容器),领任务、调本地 `/v1/tasks`、传 OBS |
| 存储层 | NFS(权重/镜像)+ OBS(产物) | 共享资产 + 产物治理 |

## 3. 组件详解

### 3.1 Gateway / API 服务(FastAPI,控制节点)
- **对外**:`POST /web/tasks`(网页)、`POST /v1/images/generations`·`/v1/images/edits`(复用 LightX2V 已有格式)、`POST /v1/videos`(新建,OpenAI 视频风格)。
- **职责**:鉴权(API Key)→ 参数校验 → 生成任务记录(Postgres)→ 入队(Redis)→ 返回 `task_id`;轮询/结果接口回传 OBS URL。
- **不直接碰 GPU**,纯接入 + 编排。

### 3.2 Redis(队列 + 节点状态)
- **任务队列**:按模型分区(如 `queue:ltx2`、`queue:wan2.2_720p_ul4`),或单队列带模型标签。
- **节点状态**:每节点心跳写 `node:<id>`(serving 的模型/拓扑、busy/free、GPU、最后心跳)。
- 作为快速调度面;持久记录落 Postgres。

### 3.3 Dispatcher(调度器)
- 循环:从队列取待派任务 → 找"**serving 该模型/拓扑且空闲**"的节点 → 派发 → 标记节点 busy。
- 视频生成 GPU 重,**单节点并发=1**(一次一条),busy/free 标志即可。
- 策略:FIFO 起步,后续可加优先级/公平队列。
- 找不到匹配节点 → 任务留队列等待(或返回"该模型无在线节点")。

### 3.4 Node Agent(每计算节点)
- **注册**:启动时把节点能力(serving 模型/拓扑、GPU 数)写入 Redis/Postgres;定时心跳。
- **领任务**:从 Dispatcher 接收派发(或从队列拉)。
- **执行**:调**本地** LightX2V server 的 `POST /v1/tasks/video` + 轮询 status(完全复用现有 API)。
- **产物**:生成完上传 OBS,回写产物元数据 + 状态。
- **与 LightX2V 容器关系**:Agent 是节点上的轻量守护进程;LightX2V 容器是推理运行时,两者同机。

### 3.5 LightX2V 运行时容器(复用)
- 就是已验收的镜像 `arm64-a100-...`。
- 单卡:`python -m lightx2v.server ...`;4卡:`torchrun --nproc_per_node=4 ...`(**直接复用 §15 验证过的命令**,不经任何调度器包装)。
- 启动配置由网页生成(见 §5)。

### 3.6 Postgres(持久状态)
- 表:`nodes`、`tasks`(参数/状态/OBS url/耗时)、`api_keys`、`models`(可服务的模型目录)。

### 3.7 Web 前端
- **运营页**:配置节点(选模型/拓扑/量化/参数 → 生成 config → 该节点 (重)启 LightX2V)、查看节点/队列状态、加节点向导。
- **生成页**:填 prompt/分辨率/帧数/seed/(i2v)图 → 提交 → 进度 → 视频内嵌播放 + OBS 下载。

## 4. 关键数据流

### 4.1 生成(网页或 OpenAI)
```
客户端 → Gateway(鉴权,建 task,入队 Redis,返 task_id)
       → Dispatcher(匹配空闲且 serving 目标模型的节点)→ Node Agent
       → 本地 LightX2V /v1/tasks/video(提交) → 轮询 status
       → 完成:Agent 上传 OBS → 回写 task(OBS url)
客户端 ← 轮询 Gateway 状态 ← 完成后拿 OBS(预签名)URL
```

### 4.2 节点配置(网页配置,不读本地文件)
```
运营在网页选 模型+拓扑+量化+参数 → Gateway 生成 config dict
   → 写临时 /tmp/web_cfg_<ts>.json(仅进程间传参,跑完即弃)
   → 该节点 docker run 起 LightX2V(单卡 python / 4卡 torchrun)+ --config_json 临时文件
   → 等 health 200 → 节点状态置为 serving <模型/拓扑>
```
> 临时 config **完全由网页输入生成**,不读 `/data/lightx2v_configs/*.json`。

## 5. "配置走网页"落地:字段映射

网页表单字段 ↔ LightX2V config 字段(草拟,需用现有 server config 校准——见 §11):

| 网页字段 | → config / 启动参数 |
|---------|--------------------|
| 模型(LTX2.3 / Wan2.2-int8) | `model_cls`(ltx2 / wan2.2_moe)、`model_path` |
| 任务(t2v / i2v / t2av) | `--task` |
| 拓扑(单卡 / 4卡 ulysses) | 启动命令:`python` vs `torchrun --nproc_per_node=4`;config 的 `seq_parallel` 等 |
| 量化(bf16 / int8) | `--config_json` 内量化字段、`*_quantized_ckpt` 路径 |
| 分辨率 / 帧数 | 默认值(也可每请求覆盖 `target_shape` / `target_video_length`)|
| 步数 / cpu_offload / attention / boundary(Wan)/ VAE tiling 环境变量(LTX) | 对应 config 字段 / 容器环境变量 |
| **红线(后端硬编码)** | Wan2.2 一律预量化 int8、**禁用在线 LoRA**(`lora_dynamic_apply=false`),避免雪花废片 |

## 6. 多节点与队列

- **节点定位策略(默认:静态绑定)**:每节点开机即被配置为 serving 某个"模型+拓扑"(如 node-A=LTX 单卡,node-B=Wan int8 4卡 720p)。Dispatcher 按模型路由。
  - 理由:切模型要 5~10min 重载,平台不宜每请求切。需要变更时由运营在网页"重配该节点"。
- **队列**:全局 Redis 队列(按模型分区);每节点并发 1;FIFO 起步。
- **各模型拓扑(已实测,2026-06-25)**:
  - **LTX2.3 = 固定单卡 bf16**(block offload,峰值 ~18.9GB)。**多卡不可用**:在 40GB A100 上 ulysses(每 rank 复制 46GB → CPU OOM)和 TP(rank0 把整 46GB 暂存单卡 GPU → GPU OOM)**均实测失败**;根因是 LightX2V 分布式加载实现(非 config 可绕过)。详见 §6.1。
  - **Wan2.2 int8 = 单卡 或 4卡 ulysses**(int8 量化后权重小、装得下;冒烟用例3 已验证 4卡 720p 干净)。
  - 单卡节点:一机多卡 = 各起独立单卡实例(提吞吐,不提单条速度)。
- **决策点 ⚠️**:静态绑定 vs 允许节点按队列动态切模型 —— 默认静态,更稳;要弹性再议。

### 6.1 LTX2.3 多卡实测结论(2026-06-25,镜像 `arm64-a100-20260625-0256-2cde4721`)

| 模式 | 结果 | 根因 |
|------|------|------|
| 单卡 + block offload | ✅ 出片(81帧:加载 215s + 文本编码 335s + DiT 69s) | 逐块流式,峰值 18.9GB |
| ulysses 4卡(seq_p_size=4) | ❌ CPU OOM,42min 后 rank 被 SIGKILL | 每 rank 复制完整 46GB → 4×=184GB 撑爆 251GB 内存 |
| TP 4卡(tensor_p_size=4) | ❌ GPU OOM,1min | `_load_weights_from_rank0` 在 rank0 GPU 上 `torch.empty` 暂存全 46GB > 40GB |
| TP 4卡 + cpu_offload | ❌ GPU OOM,1min | TP 加载标 "CUDA only",cpu_offload 不改暂存设备 |

- 上游虽已加入 ltx2 的 seq_parallel/tensor_parallel 代码与 `configs/ltx2/ltx2_ulysses.json`、`ltx2_tp.json`,但**对 22B 模型 + 40GB 卡的加载阶段直接 OOM**。
- **未来若要 LTX 多卡**:需 ① 80GB 卡(TP rank0 暂存能放下),或 ② 上游改成"每 rank 从磁盘直接加载自己分片"(避免 rank0 全模型暂存),或 ③ TP 暂存改走 CPU。**当前一律单卡。**
- 另:`tools/convert/converter.py` 仍不支持 ltx2 → **无 int8 离线量化路径**;fp8 配置存在但面向 sm_90+,A100(sm_80)无 FP8 算力。**LTX 在 A100 上只能 bf16。**

## 7. 动态加节点 + NFS

**加节点流程(向导化)**
```
1. 新机挂 NFS(权重 + 镜像 tar 或私有 registry)
2. docker load < NFS 上的镜像 tar(或从 NFS 上的 registry 拉)
3. 起 Node Agent(配置:控制节点地址、本节点要 serving 的 模型/拓扑)
4. Agent 据配置起 LightX2V 容器(model_path 指向 NFS 权重)
5. Agent 注册 + 心跳 → Dispatcher 纳入调度
```
- **NFS 内容**:`/nfs/models/...`(全部权重,各节点共享只读)、`/nfs/images/lightx2v_*.tar`(或私有 registry)。
- **IO 风险 ⚠️**:46GB 级权重从 NFS 加载可能慢(文档多次强调 SSD)。缓解:首次加载后**缓存到节点本地 SSD**,或保证 NFS 高带宽。需实测。

## 8. 产物入 OBS + 生命周期

- 生成完,Node Agent 用华为云 OBS SDK 上传产物到 bucket(key 如 `videos/<date>/<task_id>.mp4`)。
- **生命周期规则**(OBS 控制台/SDK 配置在 bucket 上):如 N 天后转低频/归档、M 天后过期删除。
- Gateway 给客户端返回 **预签名 URL**(LightX2V 请求 schema 里已有 `presigned_url` 字段,可对接)。
- 元数据(task_id→OBS key、过期时间)记 Postgres。

## 9. OpenAI 标准接口

| 接口 | 现状 | 做法 |
|------|------|------|
| 图像生成 `/v1/images/generations` `/edits` | ✅ LightX2V 已有 OpenAI 格式 | Gateway 透传/复用 |
| 视频生成 | ❌ 无 | **新建** OpenAI 视频风格端点:`POST /v1/videos`(参数 model/prompt/size/seconds/seed → 映射 LightX2V 视频任务)→ 返回 job;`GET /v1/videos/{id}` 轮询 → 完成给 OBS URL |
- 鉴权/Key/配额:Gateway 自带简单实现;若以后要多服务统一纳管,再在前面挂 new-api(可选,非必须)。

## 10. 技术选型

| 模块 | 选型 | 理由 |
|------|------|------|
| Gateway/Agent/Dispatcher | Python + FastAPI | 镜像/生态已有,与 LightX2V 同栈 |
| 队列/节点状态 | Redis | 轻、成熟、够用 |
| 持久状态 | Postgres | 任务/节点/Key 记录 |
| 前端 | 单页 HTML/JS(或 Vue 轻量) | 无重前端栈 |
| 节点通信 | HTTP(Agent ↔ Gateway/Dispatcher)+ 心跳 | 简单可控 |
| OBS | 华为云 OBS SDK(Python) | 产物存储 + 生命周期 |
| 容器编排 | 直接 docker(Agent 调) | 不引入 k8s,保持轻 |

## 12. 关键风险

1. **NFS 加载大权重的 IO 瓶颈** → 本地 SSD 缓存层。
2. **同机多卡的实例划分**(4卡整机一实例 vs 拆 4 个单卡)需结合显存实测。
3. **OpenAI 视频接口无统一标准**,需自定义 schema,注意与客户端约定。
4. **网页生成 config 与现有验证 config 的等价性**(漂移会导致出片效果不一致)。
5. **安全**:Gateway 能起容器、跑任意 config、传 OBS → 内网部署 + Key 鉴权 + 最小权限。

## 13. 分阶段路线图

| 阶段 | 内容 | 验收 |
|------|------|------|
| **P0** | 数据收集 + 单节点闭环原型:网页配置→生成 config→起 LightX2V(单卡 LTX/Wan,再 4卡)→生成→**传 OBS** | 单机网页点出片 + 落 OBS |
| **P1** | 拆出 Gateway + Redis 队列 + Node Agent;2 节点;Dispatcher 按模型路由 | 多节点排队出片 |
| **P2** | 动态加节点向导 + NFS 取权重/镜像 | 新机一键纳入 |
| **P3** | OpenAI 接口:图像(复用)+ 视频(新建);Key/配额 | 标准 API 可用 |
| **P4** | 硬化:鉴权/监控/指标/OBS 生命周期策略/失败重试 | 生产可用 |

> 每阶段都以"能跑通 + 复用已验证 LightX2V 命令"为准,避免一次性铺大摊子。
