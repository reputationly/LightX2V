# 开源视频/多模态模型选型调研 — A100 (sm_80 / ARM) on LightX2V

> 面向：在已跑通 **Wan2.2 Lightning** 和 **LTX 2.3 distill 1.1** 之后，评估"还有哪些开源模型值得在我们硬件上试"。
> 调研日期：2026-06-14。
> 配套文档：`docs/LTX-2.3-单卡A100-POC交接.md`（Wan/LTX 的实测结论、出包、offload 体系）。

---

## 0. 一句话结论

除已测的 Wan2.2 / LTX2.3 外，**最该先验的是 HunyuanVideo 1.5（蒸馏 bf16/int8）**——三大主流开源视频模型里唯一没测的，LightX2V 官方 Day0 支持、显存合身、有非 FP8 量化路径。其次是 **Wan2.2-I2V 4step 720p 新权重**（复用现有 Wan 体系）和 **SeedVR2 超分**（出片后处理链）。

LightX2V **官方在线服务**（x2v.light-ai.top）当前主推的生产级模型集：**LTX 2.3、Wan 2.2、SekoTalk、Qwen-Image、SeedVR2**。

---

## 1. 硬件甜区与选型铁律（务必先读）

我们的硬件卡得很死，选型第一优先级永远围绕这三条：

| 约束 | 后果 | 选型铁律 |
|---|---|---|
| **A100 40GB / sm_80** | 无原生 FP8（FP8 要 Hopper/Ada）；无 FP4（要 Blackwell） | **只选 bf16 或 int8 路径**，FP8/FP4 配置一律跳过或回退 |
| **ARM aarch64** | PyPI 被墙、部分 CUDA 扩展要源码编译 | 优先用 LightX2V 已原生支持的模型，复用现有镜像/出包体系 |
| **40GB/卡** | 大模型要 offload / 量化 / 多卡 | 优先有 offload + 蒸馏（少步数）配置的模型 |

> 既有结论佐证：**LTX 固定 bf16、Wan 必须预量化 int8**（见 POC 交接文档 §14.9）。同一逻辑套到任何新模型。

## 3. 落地优先级

### 第一梯队：官方背书 + 硬件合身，直接可试

#### 3.1 HunyuanVideo 1.5 — 最优先

- **是什么**：腾讯混元 8.3B DiT 视频模型（原版 HunyuanVideo 是 13B/需 47–58GB，这一代专为消费级缩身）。原生 480p/720p，super-res 到 1080p，T2V + I2V，中英双语。质量在开源里属"电影感天花板"一档。
- **LightX2V 支持**：官方 Day0 支持（2025-11-21），同卡数 **2× 提速**，支持 24GB 4090 这类低显存部署；CFG/Ulysses 并行、offload、TeaCache/MagCache 齐全。
- **蒸馏/量化**：
  - 4 步蒸馏（2025-11-24）：无需 CFG，相比 50 步约 **25× 加速**，base + FP8 两版 → `lightx2v/Hy1.5-Distill-Models`
  - 量化模型 → `lightx2v/Hy1.5-Quantized-Models`；轻量 TAE（快速 VAE 解码）→ `lightx2v/Autoencoders/lighttaehy1_5.safetensors`
- **框架现成配置**（`configs/hunyuan_video_15/`）：
  - `offload/hy15_t2v_480p_bf16.json` — 复用 LTX 的 offload 思路
  - `hunyuan_video_t2v_480p_distill.json` — 蒸馏快出片
  - `cache/*_teacache.json` — TeaCache 加速
  - ⚠️ `quant/*_fp8.json`、`fp8comm/*` — **sm_80 跑不出 FP8 加速，不优先**
- **显存**：bf16 全链路约 24–28GB，40GB 单卡轻松装下。
- **我们的走法**：**bf16 / int8 蒸馏版 + 4 步 + 单卡**，避开 FP8。先 `offload/hy15_t2v_480p_bf16.json` 冒烟 → 上 720p。

#### 3.2 SeedVR2（`seedvr2`）— 视频超分，互补型

- 官方在线服务带它。**不是替代生成**，而是把 LTX/Wan/混元 出的 480p/720p 统一超分到 1080p+。
- 建议搭成所有出片的**统一后处理链**。

### 第二梯队：Wan 生态扩展（复用现有 Wan 权重 + server 体系）

| 模型 | 输入 → 产出 | sm_80 量化 | 额外依赖 | 备注 |
|---|---|---|---|---|
| **Wan2.2-I2V 4step 720p high/low**（2026-04-20 新权重） | 图 → 视频 | int8 | 无 | 720p 高质量数据重训，细节纹理提升，**直接换权重试** |
| `seko_talk`（SekoTalk） | 人像 + 音频 → 对口型数字人 | ✅ 有 int8 配置（`seko_talk_25_int8_dist_fp8_comm.json`） | 音频处理 | 数字人/口播；多人、实时、vsr 变体齐全 |
| `wan2.2_moe_vace`（VACE） | 参考/control 条件 → 可控编辑/重绘 | ✅ **有 `configs/wan22_vace/a800/int8` 现成配置（A800=sm_80 同架构）** | 无明显新依赖 | "视频版 ControlNet + 编辑套件"，接入风险最低 |
| `wan2.2_animate` | 参考图 + 驱动视频 → 角色动画/换人 | 看 lora_dist 配置 | **需 decord（未装，要重出镜像）** | 效果最亮眼；`replace_flag` 切 animation/换人两模式；720×1280/77帧 |

### 第三梯队：图像类（业务要出图再上）

- **Qwen-Image-2512 / Qwen-Image-Edit-2511**：官方 Day0，8 步 CFG/步数蒸馏 + FP8（FP8 加速在 sm_80 享受不到，用 bf16/蒸馏 lora）。在线服务带。
- `z_image` / `longcat_image` / `flux2`（klein/dev）/ `bagel`：框架支持，纯图像，按需。

## 4. 明确别碰（硬件不匹配 / 小众）

| 项 | 原因 |
|---|---|
| **Wan2.1 NVFP4**（2025-12-22） | FP4 要 Blackwell（5090/B 系），A100 sm_80 跑不了 |
| **所有 FP8 加速配置** | sm_80 无原生 FP8，回退或无收益。一律选 bf16/int8 |
| **WorldMirror 2.0 / worldplay / matrix_game** | 世界模型/可交互，官方测速基于 H100，场景小众 |

---

## 5. 关于 NSFW（内容安全）

- Wan / HunyuanVideo / LTX 官方权重 + LightX2V 框架**本身不带 NSFW 过滤器**，纯推理引擎。官方版对露骨内容生成能力有限（未专门训练）。
- 社区第三方 NSFW 微调 LoRA 存在，但不在我们现有官方权重范围。
- **合规方向**：若需求是给线上服务**加 NSFW 拦截/审核**（输入 prompt 过滤 + 输出帧审核），属正当防御性工作，可单独设计。

---

## 6. 建议试用顺序

1. **HunyuanVideo 1.5 蒸馏 bf16/int8（4 步，单卡）** — 最优先，新模型、官方背书、显存合身
2. **Wan2.2-I2V 4step 720p 新权重** — 直接换上，复用现有 Wan 体系
3. **SeedVR2** — 统一超分后处理链
4. 业务要出图再上 **Qwen-Image / Qwen-Image-Edit**

> 下一步可照 POC 交接文档 §14.4 的格式，先给 HunyuanVideo 1.5（bf16 蒸馏单卡）写 `start_hy15.sh` + 验证脚本，直接在 A100 冒烟。
